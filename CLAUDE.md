## 2. Что строим

- Онлайн-запись к бьюти/сервис-мастерам. Рынок: Узбекистан / СНГ. UI на русском. Mobile-first.
- **Двухролевое** приложение: мастер и клиент в одном продукте, роль выбирается на онбординге.
- **Стратегия — НЕ маркетплейс сразу.** Ядро: инструмент для мастера (планировщик + мини-CRM + публичная страница записи + напоминания). Полезен при нуле клиентов в системе — обходит холодный старт. Каталог / поиск / рейтинги — слой поверх, позже.

**Отличия от конкурента (Masters by JamSoft):**
- локальные узбекские платежи (Payme / Click) вместо ЮKassa (у них «оплачивать сложно» в UZ);
- приватность и локализация ПДн (у них данные текут в Yandex/РФ);
- единое приложение вместо трёх (мастер/клиент/салон);
- **Telegram Mini App** как клиентская поверхность — у них нет, бесплатная телеграм-дистрибуция;
- чистый OTP-онбординг (у них жалобы на залипший вход);
- реалтайм-чат на Supabase.

## 5. Модель данных и организации

**Организация (tenant) — единая модель для соло и салона.** `organization(type: solo | salon, subscription_id)`. Соло-мастер = организация из одного человека (он и owner, и master). Салон = организация с админом и N мастерами. Один код-путь, не два; соло легко «дорастает» до салона.

**Участники и роли.** `membership(org_id, user_id, role: owner | admin | master, status: invited | pending | active | left)`. owner/admin — администрация салона (полный доступ, все поверхности); master — работает, ведёт своё расписание. MVP: у мастера одна активная организация; схема готова к нескольким (freelancer в 2 салонах).

**Подписка — на уровне организации, не мастера.** `subscription(org_id, plan, seats, status)`. Салон покупает одну подписку, под ней работают все мастера (seats = число мастеров). Соло — свой план (free/solo).

**Профиль и данные мастера.** `master_profile(user_id, cover_type: color|photo, cover_color=бренд-акцент, visible_in_search: bool=false)`. `service`, `availability`, `booking(status)`, `client`(ПДн!), `payment`, `gallery` (портфолио) — принадлежат мастеру (`master_id`) внутри организации (`org_id`).

**Флоу регистрации — три опции на входе: Клиент / Мастер / Салон.**
1. **Клиент** → клиентский аккаунт (не входит в штат организации).
2. **Мастер (соло)** → `organization(type=solo)` + `membership(owner+master)`.
3. **Салон** → затем выбор **Админ** или **Мастер**:
   - Админ → создаёт салон (owner/admin новой `organization(type=salon)`) или принимает приглашение админом существующего.
   - Мастер → присоединяется к существующему салону (код/ссылка/поиск) → `membership(status=pending)` → админ подтверждает (`status=active`), и расписание мастера открывается салону.
   Без approve мастер не привязывается; без заявки/приглашения салон не поглощает.

**Telegram-подключение мастера.** `telegram_link(user_id, tg_user_id)` через deep-link токен (/start). Бот шлёт уведомления о записях; Mini App даёт мастеру редактировать расписание. Нативное приложение мастеру не обязательно — Telegram достаточно.

**Поверхности — без ограничений по ролям.** Клиент, мастер и админ — каждый может пользоваться вебом, приложением (iOS/Android) и Telegram (Mini App + бот). Никого не запираем в одной поверхности; бэкенд один, клиент выбирает удобное.

**RLS теперь по «организация + роль»:** строки несут `org_id` и `master_id`. Роль master → только свои строки (`master_id = me`); роль admin/owner → все строки своей организации. Ключ — membership + роль.

**Правило видимости (без изменений):** публичная ссылка/QR на запись работают ВСЕГДА (ядро продукта). `visible_in_search` управляет только видимостью мастера в поиске для посторонних.

**Решить до постройки этой фичи (дефолты — можно менять):**
- Мультичленство: мастер в >1 салоне? Дефолт — схема M:N, MVP одна активная организация.
- Клиенты при уходе мастера: чьи? Дефолт — клиенты принадлежат мастеру (переносимы), салон видит брони/расписание, но не забирает базу. Зона владельца.
- Что видит админ: расписание + брони + выручку по мастеру; личные заметки мастера — приватны. Зона владельца.
- Оплата: подписка салона — по местам (seats); кто платит за соло — зона владельца.

## 12. Кросс-каттинг (детали — в `app-spec.md`)

- **Аккаунт = телефон (OTP), один на номер.** Под ним несколько профилей: клиент и/или мастер (solo-org) и/или членства в салонах. Один человек по одному номеру может быть и клиентом, и мастером. **Тумблер** (главный экран + меню) переключает Клиент ↔ Мастер/Админ без релогина. Выбор роли, которой ещё нет → создаётся новый профиль под тем же аккаунтом.
- **Данные на бэкенде, не на устройстве.** Смена устройства = вход по номеру, всё (профили, история, брони, клиенты) подгружается автоматически. Локальная БД — только кэш.
- **Языки:** база **English**; en / ru / uz (uz-Latn) на старте. Первый вход — язык устройства, если он en/ru/uz, иначе English. Меняется в настройках, хранится на бэкенде. Все строки — через i18n, не хардкодить.
- **Тема:** по устройству (system light/dark) + ручной оверрайд в настройках.
- **Телефон-инпут:** международный, плейсхолдер `+998`, определяет страну по коду и показывает флаг; хранение E.164 (libphonenumber).
- **Календарь:** выборочная синхронизация — подключить внешний календарь (Google/Apple), выбрать что синкать. MVP — пуш брони в календарь.
