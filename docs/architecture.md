# Архитектура: готовность к мульти-рынку

Цель: запуск в Узбекистане, затем Центральная Азия, ЛатАм, Африка,
Юго-Восточная Азия — **без переписывания ядра**. Эти правила обязательны
для всего кода (web, mobile, backend) с первого дня.

## Принципы

### 1. Реестр рынков — единственный источник странового

Всё, что зависит от страны, живёт в реестре рынков
(`web/lib/markets.ts`, позже — таблица `markets` в БД):
валюта, телефонный код, часовой пояс по умолчанию, языки, канал
напоминаний, платёжные провайдеры. Выход на новый рынок = новая
запись в реестре + переводы, ноль правок в логике.

### 2. Никаких строк интерфейса в коде

Все тексты UI — через слой переводов (`web/lib/i18n.ts`).
Языки сейчас: ru, uz, en, es. Тексты, которые пишет мастер
(названия услуг, bio), не переводятся — это его контент.

### 3. Деньги

- Хранение: целое число в минимальных единицах валюты + код валюты ISO 4217.
- У каждого мастера своя валюта (поле `masters.currency`).
- Форматирование только через `Intl.NumberFormat` / реестр рынков.
- Никогда не хардкодить «сум», знаки валют и разделители разрядов.

### 4. Телефоны

- Хранение: E.164 (`+998901234567`), без пробелов и скобок.
- Валидация и плейсхолдеры — из конфига рынка, не из регулярки в компоненте.

### 5. Время

- В БД только `timestamptz` (UTC).
- У мастера есть `masters.timezone` (IANA). Слоты, «сегодня», напоминания
  считаются в поясе мастера, а не клиента и не сервера.

### 6. Уведомления — абстракция канала

Telegram — это канал рынка UZ/СНГ, а не платформы. На других рынках:
WhatsApp (ЛатАм, Африка, ЮВА), Zalo (Вьетнам), LINE (Таиланд), SMS (fallback).
Код отправки работает с интерфейсом «канал», конкретный провайдер
выбирается по конфигу рынка/мастера.

### 7. Платежи за подписку — абстракция провайдера

Payme/Click/Uzum — провайдеры рынка UZ. Дальше: Kaspi (KZ),
Mercado Pago (ЛатАм), M-Pesa/Flutterwave (Африка), Xendit/Midtrans (ЮВА),
Stripe (глобальный fallback). Биллинг работает с интерфейсом
«провайдер подписки»; статус подписки в БД един для всех провайдеров.

## Поправки к схеме БД из брифа

Дельта к `docs/project-brief.md`:

```sql
markets (
  id text PK,                 -- 'UZ', 'KZ', 'MX', ...
  default_locale text,
  currency text,              -- ISO 4217
  timezone text,              -- IANA, дефолт
  phone_dial_code text,
  reminder_channel text
)

users (
  ...
  phone text UNIQUE,          -- строго E.164
  lang text                   -- 'ru'|'uz'|'en'|'es', расширяемо
)

masters (
  ...
  market_id text REFERENCES markets,
  currency text,              -- наследует от рынка, можно переопределить
  timezone text               -- IANA
)

services (
  ...
  price int                   -- минимальные единицы валюты мастера
)

notifications (
  ...
  channel text                -- 'telegram'|'whatsapp'|'sms'|'line'|'zalo'
)

subscription_payments (
  id uuid PK,
  master_id uuid REFERENCES masters,
  provider text,              -- 'payme'|'click'|'uzum'|'stripe'|...
  amount int,
  currency text,
  paid_at timestamptz
)
```

## Что сознательно НЕ делаем сейчас

- Не подключаем тяжёлые i18n-библиотеки — словаря достаточно,
  миграция на next-intl механическая, если понадобится.
- Не делаем мультивалютные цены у одного мастера — мастер работает
  в одной валюте.
- Не строим автоматическое определение локали по IP — язык страницы
  мастера задаёт сам мастер (его клиенты — местные).
